Before you start

Bring your content, not a finished setup

You do not need to install anything before the workshop. Bring your laptop, your content, and a pen. We will do the sketching and setup together during the workshop.

Bring this to the workshop

  1. Bring your laptop and charger.
  2. Bring paper and a pen for the first workshop step.
  3. You do not need to install anything before the workshop. We can set up VS Code and OpenCode together.
  4. If you already have Git or a GitHub account, that helps, but it is not required in advance.
  5. Keep your text, images, and links in one folder so they are easy to access during the workshop.

Please prepare your content before the workshop

  • Your name or artist name
  • One short introduction, 3 to 5 sentences
  • Project titles and short descriptions
  • Links: Instagram, email, website, CV, event page
  • 2 to 8 images you are allowed to publish

The goal

Keep it simple. Finish something real.

This workshop is for beginners

You do not need coding experience. We use plain HTML, CSS, and a little JavaScript only when it helps. We do not use React, single-page apps, or component-heavy abstractions in the main workshop path.

One real file per real page

If your sketch shows several pages, the website should usually have several HTML files. That keeps the project intuitive and easy to edit later.

Plan first with OpenCode

OpenCode helps most when it reads your plan.md, asks design questions, and agrees on the structure before it starts writing files.

Publish early, then improve

Try to get a small first version online early in the workshop. After that, preview locally in VS Code, improve step by step, and push updates at a few checkpoints.

Basics

What each tool does

The workflow is simpler if each tool has one clear role during the workshop.

What is VS Code?

VS Code is where you open your website folder, inspect files, preview changes locally, and use Source Control to commit and push your work.

What is the terminal?

The terminal is the text-based panel inside VS Code where you run commands. In this workshop, that is where you launch OpenCode with opencode.

What is OpenCode?

OpenCode is the coding agent we will all use. It can read your plan.md, ask questions in Plan, create files, and explain what it changed. You should still read the changes and test the result.

What is GitHub?

GitHub stores your project online, keeps a history of changes, and is where you publish the public site with GitHub Pages.

Setup

Run OpenCode inside VS Code

OpenCode lives in the integrated terminal. VS Code stays the place where you read files, preview pages, and push changes. If nothing is installed yet, that is fine. We can do the setup in the workshop.

Install OpenCode

If OpenCode is not installed yet on the day of the workshop, use the official install command here. Students do not need to do this in advance.

Install command

curl -fsSL https://opencode.ai/install | bash

Official docs: opencode.ai/docs

Open the VS Code terminal and start OpenCode

Open your project folder in VS Code, then use View → Terminal. Start OpenCode in that terminal. If you still need to sign in, connect your account inside the app before you continue.

Start OpenCode

opencode
/connect

The first line runs in the terminal. The second line is a command you can type inside OpenCode if you still need to connect your account.

How the tools split the work

  • Use VS Code to open files, inspect changes, and preview locally.
  • Use OpenCode in the terminal to plan, ask questions, and implement changes.
  • Use GitHub to publish the public site.
  • Use Source Control in VS Code for commits and pushes.

Workshop flow

Follow these steps in order

Step 1

Sketch your website on paper

We do this together as the first workshop step. Start away from the computer. Sketch the pages, subpages, navigation, and the main blocks on each page. Keep the first version small enough to finish in one workshop.

  • Write the purpose of the website in one clear sentence.
  • Choose the pages or subpages you want.
  • Sketch the rough layout for each page.
  • Mark the main text, images, and links for each page.
  • Draw the navigation between pages.
  • Note the visual mood you want.

Step 2

Create the folder and write plan.md

Open a new website folder in VS Code. Turn your paper sketch into a clear plan.md. Then copy AGENTS.template.md from this workshop repo into your own project and rename it to AGENTS.md.

Starter file list

AGENTS.md

plan.md template

# Audience

# Pages / subpages

# Sections on each page

# Content and assets

# Visual direction

# Must-have features

# Later ideas

Step 3

Use OpenCode Plan before coding

Open the integrated terminal in VS Code, start OpenCode, switch to Plan, and let it read plan.md before it creates any files. Answer its design questions so the structure becomes clear first.

Ask OpenCode in Plan:

Read my plan.md. I am a beginner. Please ask me the most important design and structure questions before writing code. I want plain HTML files for real pages, not React or a single-page app.

Step 4

Build the approved plan with plain files

Once the plan is clear, ask OpenCode to build the first version. If your sketch has several pages, the project should have several HTML files.

Ask OpenCode to build:

Based on plan.md and AGENTS.md, build the first version of this site. Use plain HTML, CSS, and vanilla JavaScript only when needed. Use one real HTML file per page and keep the code easy for a beginner to edit.

Step 5

Publish the first version to GitHub

Create a public repository on GitHub. In VS Code, publish your folder to that repository or use Source Control to commit and push your files.

  1. Create a new public repository on GitHub.
  2. Commit your files in VS Code.
  3. Push them to GitHub.

This first push can be small. The goal is to get online early, not to be finished.

Step 6

Turn on GitHub Pages

Open your repository on GitHub. Go to Settings → Pages. Under Build and deployment, choose Deploy from a branch, then select your main branch and the root folder.

After a short wait, your site will be live at a GitHub Pages URL, usually https://yourusername.github.io/your-repository-name/.

Step 7

Preview locally with Live Server

Install the Live Server extension for VS Code by Ritwick Dey. It adds a simple local web server and a Go Live button so you can preview changes while you review what OpenCode changed.

Important: this preview is only on your laptop. GitHub Pages is the public website.

Step 8

Iterate with OpenCode and push checkpoints

Improve one thing at a time: content, navigation, images, spacing, colors, or mobile layout. Push your work after each useful checkpoint.

Ask OpenCode for follow-up work:

Read my current files and help me improve one thing at a time. Keep the same HTML file structure, do not switch to React, and explain the change before I push it.

Prompting tips

How to work well with OpenCode

Good prompt habits

  • Say that you are a beginner.
  • Tell OpenCode to read plan.md and AGENTS.md first.
  • Ask for one small change at a time.
  • Remind it to keep one HTML file per real page.
  • Ask OpenCode to explain the file structure in simple words.
  • Preview locally before you push.

Prompt examples

Useful prompts

Read my `plan.md` and ask me the most important design questions before writing code.

Based on `plan.md` and `AGENTS.md`, build the first version as plain HTML, CSS, and JavaScript.

Explain the files you created in simple language.

Add a new Projects page and update the navigation. Keep one HTML file per page.

Improve the mobile layout, but keep the code easy for a beginner to edit.

Check the current site for mistakes and tell me what to fix first.

Publishing

Final checklist for GitHub Pages

  1. Your homepage file is named index.html.
  2. Every page in the navigation has a matching .html file.
  3. Your links between pages work locally before you publish.
  4. Your repository is public.
  5. Your files are pushed to the default branch.
  6. GitHub Pages is enabled in repository settings.
  7. You preview locally with Go Live or another local server while editing.
  8. You wait 1 to 5 minutes after publishing and test the live link on your phone.

Sketch it. Plan it. Build it. Publish it.
